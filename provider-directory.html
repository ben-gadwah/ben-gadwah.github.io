Provider Directory

(InterSystems)

Provider Directory is an application that allows hospitals to sort providers, where they're working, and which insurances they take. The app has a data engine which collects information from sources and create a "master" record. An administration user would be able to create definitions for which source information survived in the record and for how long. Other data steward users would also be able to make edits that overrode contributing sources. This data steward was someone who uses the Provider Directory daily to assure all of the records were accurate and up to date. Their workflow included navigating through providers (through some indicators such as their address or NPI number), check the related locations and organization to that provider, and verify that the information was accurate.

Revamped Navigation

Finding A Record

  • Simple stacked tree for organization
  • Cards with sample provider info for navigation
  • Cleanup actions and collapsible headers for clean workspace

Record Display Features

  • Magellan navigation to show available sections
  • Important provider information available in the header
  • Easy navigation to other aspects of the record (Contributing Records, History, Edit, etc)

Master Record Comparison

  • Master record presented in a shade of blue
  • Contributing fields are also blue to indicate their connection to the master
  • Data is organized into readable columns and fields are aligned

Viewing Past Record Versions

  • Magellan navigation across the top for sources out of view
  • Expand historic master records from transactions
  • Relevant time stamp with absolute time available via tooltip

Collaboration

I initiated and facilitated alignment meetings with all members of the team. It took many team meetings to align our knowledge and expertise of the workflows to design smart solutions. It was amazing to see members of the team from development to data quality collaborate creatively.

Design

Leading with design is new to the Healthcare tech process. The fancy word is “Pragmatic Interoperability” - designing around workflows providing care. Our goal for the Master Browser was to increase simplicity navigating through providers, clinics, and insurance networks.

From these alignment meetings collecting the team’s expertise and creative solutions I worked closely with the development team. I set out and created rough wireframes and sketches. We had created SCSS variables for most of the design elements and were able to incorporate new ideas quickly and easily. Before merging the code with the master we verified these updates with the Product team and sought key stakeholder buy-in.

Usability Test

For the usability test I provided the script, setup, moderating, as well as parsed the findings. We had designed a smart way through records with cards and wanted to test our assumptions. In our testing sessions we learned a few important things: users were able to navigate through the Master Browser easily, but more complicated pages such as History were not as clear to navigate. The side navigation panel collapsed with an animation, but users were not able to recall where the panel went and became confused.

Testing Method

  • Tests were conducted with current Provider Directory clients in Rhode Island - 5 participants who use the current system daily
  • Each test was conducted with one participant accompanied by an InterSystems moderator and note taker, with an iFactory facilitator on hand
  • Participants used a basic prototype of the new Provider Directory interface
  • Moderator guided participants through a series of tasks using the prototype
  • Participants were asked to think out loud as they worked, and provide feedback about what they were seeing, what they were looking for, and whether or not what they were using was working well
  • Task related to the Master Browser mainly. Some participants were able to give feedback on the Link Management page
  • Each session lasted approximately 30 minutes

Future

Provider Directory is still being developed, but the team has embraced the need for good design and a solid user experience.
